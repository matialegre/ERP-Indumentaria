Resumen Ejecutivo

Visión general del sistema y sus principios
▼

🎯 ¿Qué estamos construyendo?

Un ERP SaaS multi-tenant y multi-industria diseñado para operar con la misma robustez con conexión a internet o completamente offline. El sistema está pensado para empresas con múltiples puntos de venta físicos donde la conectividad no es confiable.

📡 ¿Por qué Offline-First?

En el interior de Argentina, muchos locales comerciales enfrentan cortes de internet frecuentes. Un sistema que depende de la conexión para vender simplemente no es viable. Nuestro enfoque: el dispositivo del punto de venta es autónomo. Opera, vende, factura y registra movimientos sin depender del servidor central. Cuando recupera conexión, sincroniza automáticamente.

🏭 Industrias soportadas

👕
Indumentaria / Outdoor

Talles, colores, temporadas. SKU compuestos. Gestión de colección.

🏪
Kiosco / Almacén

Lotes, vencimientos, FIFO. Control sanitario. Alta rotación.

🔧
Taller Mecánico

Número OEM, aftermarket. Compatibilidad vehicular. Órdenes de trabajo.

🔩
Ferretería / Construcción

Bultos, pallets, ubicación estante. Unidades fraccionarias. Remitos.

⚡ Principios de Diseño

  • Offline-First: Toda operación funciona sin conexión. El server es la fuente de verdad, pero el local es autónomo.
  • Multi-Tenant: Cada empresa opera en su propio espacio aislado (company_id en cada tabla).
  • Extensibilidad: Los campos de variante se activan según el rubro. Sin rigidez.
  • Idempotencia: Toda operación offline lleva un offline_id UUID. El server rechaza duplicados.
  • Auditoría: Todo movimiento registra usuario, dispositivo, timestamp, y estado de sincronización.

🔄 Flujo de Datos

1
DESCARGA (Server → Local) Catálogos se sincronizan cada 5 minutos vía syncAllCatalogs(). Se descargan productos, stock, proveedores y locales al IndexedDB. Usa timestamp de última sincronización para delta updates.
2
OPERACIÓN OFFLINE Venta, ajuste de stock o ingreso se guarda en pendingOps + offlineSales en IndexedDB. El stock local se decrementa inmediatamente vía decrementLocalStock(). Se genera recibo HTML offline para el cliente.
3
SUBIDA (Local → Server) flushPendingOps() envía las operaciones encoladas cuando detecta conexión. Orden FIFO. Reintentos hasta 5x con backoff exponencial. Tras 5 fallos → marca FAILED para revisión manual.
4
CONFIRMACIÓN El server procesa la operación, actualiza la base de datos central, y retorna respuesta con IDs definitivos. El cliente marca la operación como SYNCED y limpia de pendingOps. Si hay conflicto → se registra para resolución.

Resolución de Conflictos

Estrategias para cada escenario de conflicto offline
▼

Este es el problema central de un sistema offline-first. Cuando dos dispositivos operan sin conexión simultáneamente, los conflictos son inevitables. La clave es tener una estrategia clara para cada tipo de conflicto.

Escenario Ejemplo Concreto Estrategia de Resolución
Mismo producto, dos locales offline Local 1 vende la última campera M azul. Local 2 también vende "la última" porque su stock local aún dice 1. Last-Write-Wins + Alerta de stock negativo.
Server acepta ambas ventas. Stock central queda en -1. Se genera alerta STOCK NEGATIVO al admin. La venta NO se rechaza (el cliente ya se fue con el producto). Admin decide: pedir reposición o transferir desde otro local.
Mismo producto, mismo local, dos cajas Caja 1 y Caja 2 del mismo local venden el último item simultáneamente. Partición por caja.
Cada caja decrementa su propia copia local de stock. Al sincronizar, el server consolida ambas ventas. Posible resultado negativo → genera alerta. Resolución idéntica al caso anterior.
Ajuste de inventario vs. venta Admin hace ajuste "+5" mientras el cajero vendió 3 unidades offline. Timestamps + merge cronológico.
Ajuste con timestamp T1, ventas con timestamps T2, T3, T4. Server aplica en orden cronológico: +5, -1, -1, -1 = resultado correcto +2. El orden temporal garantiza consistencia.
Precio cambiado mientras offline Admin cambia precio de $10.000 a $12.000. Cajero offline sigue vendiendo a $10.000. Venta vale con precio del momento.
El precio queda grabado en SaleItem.unit_price al momento de la venta. No se retroactiva. Al sincronizar, el server acepta el precio original del momento offline. Principio: el cliente pagó ese precio.
Transferencia entre locales Local A envía 10 unidades a Local B. Local B no confirma porque está offline. Estados de transferencia.
ENVIADO (A decrementó stock) → RECIBIDO (B confirma e incrementa stock). Si B está offline, queda en ENVIADO. Timeout 72h → alerta al admin. Admin puede forzar confirmación.
AFIP caído durante emisión Factura emitida offline sin CAE. AFIP sigue caído al reconectarse. Cola AFIP con reintentos.
Factura se guarda con status PENDING_CAE. Cola intenta cada 5 min, escalando a 2h. Máximo 72h de reintentos. Después → alerta manual. El cliente recibió un ticket provisorio con UUID. Alternativa: usar CAEA (CAE Anticipado) pre-solicitado.

📐 Regla de Oro

"Nunca rechazar una venta que ya ocurrió en el mundo real."

Si el cliente se fue con el producto y pagó, la operación es válida sin importar el estado del sistema. Los conflictos se resuelven después, no bloqueando la venta. El sistema registra, alerta, y el humano resuelve.

Agotó reintentos. Requiere intervención manual del administrador.

🔐 CAEA — Modo Offline Total

Cuando un local sabe que estará offline (zona rural, evento), puede pre-solicitar un CAEA (Código de Autorización Electrónico Anticipado) a AFIP. Este CAEA tiene una vigencia de 15 días y permite facturar sin conexión. Al reconectar, se informan los comprobantes emitidos.

Flujo CAEA: Solicitar CAEA → Facturar offline con CAEA → Al reconectar → POST /afip/caea/inform → AFIP registra comprobantes → ✅

Casos Borde Críticos

Escenarios de fallo y sus resoluciones
▼
🔴 CRÍTICO: Stock negativo por ventas offline simultáneas

Escenario: Dos locales venden el último item del mismo producto offline. Ambos creen tener stock = 1.

Resolución: Aceptar ambas ventas. Stock central queda en -1. Se genera alerta automática "STOCK NEGATIVO" al administrador. La venta NO se rechaza porque el cliente ya se fue con el producto.

Prevención: Configurar min_stock por local como reserva de seguridad. Sync más frecuente en productos de alta rotación.

🔴 CRÍTICO: AFIP caído más de 72 horas

Escenario: Las facturas sin CAE se acumulan en la cola. AFIP no responde por días.

Resolución: Usar CAEA (CAE Anticipado) solicitado con antelación. El CAEA permite facturar sin conexión a AFIP durante 15 días.

Fallback: Emitir como Ticket (tipo C) que no requiere CAE inmediato para monotributistas/consumidor final.

🟡 ALTO: Precio desactualizado offline

Escenario: Admin actualiza precio de $10.000 a $12.000 en el server. Cajero offline sigue vendiendo a $10.000.

Resolución: El precio en SaleItem.unit_price es definitivo. El cliente pagó ese precio; no se retroactiva. El server acepta el precio del momento offline.

Prevención: Sincronización de precios más frecuente (cada 1 minuto en lugar de 5). Notificación push vía Service Worker cuando cambian precios.

🟡 ALTO: Transferencia nunca confirmada

Escenario: Local A envía mercadería a Local B. Local B nunca confirma recepción (offline, olvido, o pérdida).

Resolución: Timeout de 72 horas → se genera auto-alerta al admin. Admin puede forzar confirmación o revertir la transferencia. El stock queda "en tránsito" visible en el dashboard.

🟢 MEDIO: Número de factura duplicado offline

Escenario: Dos dispositivos del mismo local generan facturas con el mismo número correlativo.

Resolución: El campo offline_id (UUID v4) previene duplicados en el server. El número definitivo de factura es asignado por el servidor al sincronizar. El número offline es provisional.

🟢 MEDIO: Dispositivo perdido con ventas no sincronizadas

Escenario: Tablet robada o rota que tenía pendingOps sin sincronizar.

Resolución: Auditoría de dispositivos con device_id. Dashboard muestra último sync por dispositivo. Las ventas con ticket offline impreso son prueba fiscal válida. El admin puede reconstruir operaciones desde los recibos impresos.

FASE 1: Stock Multi-Local
Dependencias: ninguna — punto de partida
  1. Crear tabla stock_by_local + migración Alembic
  2. API endpoints de stock por local (consulta, filtros)
  3. Transferencias entre locales (crear, confirmar, cancelar)
  4. Frontend: selector de local en StockPage + vista de stock por local
  5. Sync: store IndexedDB catalogStock segmentado por local
FASE 2: Variantes Multi-Rubro
Dependencias: Fase 1 completa
  1. Extender ProductVariant con campos multi-rubro (lote, OEM, vencimiento, etc.)
  2. Configuración de rubro por empresa (CompanyModule)
  3. Frontend: formularios dinámicos según rubro de la empresa
  4. Alertas automáticas de vencimiento (Kiosco/Alimentos)
  5. Buscador por compatibilidad vehicular (Taller Mecánico)
FASE 3: Facturación Robusta
Dependencias: Fase 1 completa
  1. Extender modelo Sale con campos AFIP + medios de pago
  2. Soporte múltiples medios de pago (efectivo + tarjeta + transferencia)
  3. Numeración automática por punto de venta AFIP
  4. Cuenta corriente de cliente (saldo, pagos parciales)
  5. Frontend: POS mejorado con selector de medio de pago
FASE 4: AFIP Integration
Dependencias: Fase 3 completa
  1. Instalar pyafipws + configurar certificados
  2. Crear tablas afip_config + afip_queue
  3. Endpoint /sales/{id}/authorize con WSFE (FECAESolicitar)
  4. Cola de reintentos como background task (asyncio / Celery)
  5. Solicitud de CAEA para modo offline total
  6. Frontend: indicador de estado factura (con/sin CAE, pendiente, error)
FASE 5: Sync Avanzado
Dependencias: Fase 1 + Fase 3 completas
  1. Agregar offline_id UUID en todas las tablas operativas
  2. Deduplicación server-side por offline_id (UPSERT)
  3. Resolución de conflictos por timestamp + merge cronológico
  4. Alertas automáticas de stock negativo post-sync
  5. Dashboard de sync para MegaAdmin (estado dispositivos, ops pendientes)
  6. Auditoría de dispositivos (último sync, ops fallidas, estado)
FASE 6: Módulos Nuevos
Dependencias: Fase 2 completa
  1. Órdenes de Trabajo — OT para Taller Mecánico (crear, asignar, cerrar, facturar)
  2. Archivo Digital — clasificación automática de documentos (remitos, facturas, NC)
  3. Cuenta Corriente de Cliente — saldos, movimientos, límites de crédito
  4. POS Táctil Optimizado — interfaz fullscreen para tablets en mostrador
