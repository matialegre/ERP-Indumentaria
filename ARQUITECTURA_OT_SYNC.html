01 Módulo Órdenes de Trabajo (OT)

Ciclo de vida completo de una Orden de Trabajo, desde la recepción del equipo hasta el cierre post-facturación.

📥 RECEPCIÓNIngreso del equipo
🔍 DIAGNÓSTICOEvaluación técnica
💰 PRESUPUESTOCotización al cliente
⏳ APROBACIÓNEsperando respuesta
Cliente decide
✅ APROBADO
🛠️ EN EJECUCIÓNTrabajo en curso
🔎 CONTROL CALIDADVerificación final
📦 ENTREGARetiro por cliente
🧾 FACTURACIÓNGenerar comprobante
✔️ CERRADOOT finalizada
❌ RECHAZADO
🚫 CANCELADOSin cargo / devolución
⚠️ CANCELADO puede alcanzarse desde cualquier estado previo a CERRADO. Si hay stock consumido, se re-incrementa automáticamente.

Detalle por Estado

📥 RECEPCIÓN

Datos requeridos: Cliente (nombre/tel/email), descripción del equipo, fotos de ingreso, número de serie (opcional), local de recepción.

Triggers: Genera número de OT auto-incremental (OT-2026-XXXX). Notificación al mecánico asignado.

Roles: ADMIN, LOCAL, VENDEDOR

🔍 DIAGNÓSTICO

Datos requeridos: Checklist de diagnóstico, fallas encontradas, repuestos necesarios estimados, horas estimadas.

Triggers: Asigna mecánico. Puede agregar items estimados (sin decrementar stock aún).

Roles: ADMIN, DEPOSITO (mecánico)

💰 PRESUPUESTO

Datos requeridos: Items con precios (repuestos + mano de obra + servicios externos), total estimado, vigencia del presupuesto.

Triggers: Notificación al cliente (WhatsApp futuro). PDF presupuesto generado.

Roles: ADMIN, ADMINISTRACION

⏳ APROBACIÓN

Datos requeridos: Respuesta del cliente (aprobado/rechazado), método de confirmación (presencial, telefónico, WhatsApp).

Triggers: Si aprobado → EN EJECUCIÓN. Si rechazado → CANCELADO. Registro de quién aprobó y cuándo.

Roles: ADMIN, LOCAL, VENDEDOR

🛠️ EN EJECUCIÓN

Datos requeridos: Items reales usados (repuestos consumidos, horas reales). Notas del mecánico.

Triggers: Stock se decrementa al agregar items tipo REPUESTO. Registro de horas de trabajo.

Roles: ADMIN, DEPOSITO (mecánico)

🔎 CONTROL CALIDAD

Datos requeridos: Checklist de calidad completado, fotos post-reparación, firma del responsable.

Triggers: Marca OT como lista para entrega. Notificación al cliente.

Roles: ADMIN, DEPOSITO

📦 ENTREGA

Datos requeridos: Fecha/hora de retiro, firma del cliente (digital), observaciones de entrega.

Triggers: Registra entrega física. Habilita facturación.

Roles: ADMIN, LOCAL, VENDEDOR

🧾 FACTURACIÓN

Datos requeridos: Tipo de comprobante, datos fiscales del cliente, medio de pago.

Triggers: Genera Sale + SaleItems automáticamente desde OT items. Envía a cola AFIP.

Roles: ADMIN, ADMINISTRACION

✔️ CERRADO

Datos requeridos: Factura emitida y pagada (o excepción autorizada).

Triggers: OT pasa a solo lectura. Se incluye en reportes. Historial completo disponible.

Roles: ADMIN, ADMINISTRACION

🚫 CANCELADO

Datos requeridos: Motivo de cancelación, autorizado por.

Triggers: Si status era ≥ EN_EJECUCIÓN: stock se re-incrementa. OT pasa a solo lectura.

Roles: ADMIN

03 Integración OT ↔ Stock

Flujo de decremento/incremento automático de stock según el ciclo de vida de la OT.

1

Mecánico agrega repuesto a OT

Al crear un work_order_item con item_type = REPUESTO, se selecciona un product_variant_id y se indica la cantidad.

2

Verificar estado de la OT

Si status >= EN_EJECUCIÓN: el stock se decrementa inmediatamente. Se crea un StockMovement tipo EGRESO con referencia a la OT.

Si status < EN_EJECUCIÓN (presupuesto): NO se decrementa — es solo estimación.

3

Transición a EN_EJECUCIÓN

Al aprobar la OT, todos los items tipo REPUESTO que no tengan stock_decremented = true se decrementan automáticamente en batch.

4

Cancelación con stock consumido

Si la OT se cancela y hay items con stock_decremented = true: se genera un StockMovement tipo INGRESO (re-incremento). El campo stock_decremented se resetea a false.

5

Operación offline

Si el dispositivo está offline, el decremento se guarda como un evento en el outbox de IndexedDB. Al reconectar, se sincroniza con el servidor aplicando el PN-Counter CRDT.

6

Conflicto: stock negativo

Dos mecánicos usan el mismo repuesto offline → ambos decrementan → stock queda negativo en servidor.

Resolución: Se genera alerta. Admin decide: reordenar, transferir de otro local, o ajustar. Las OTs NO se anulan (el trabajo ya se hizo).

💡 Regla clave: El stock nunca bloquea una OT. Si stock llega a 0 o negativo, el mecánico puede seguir trabajando. La alerta es para el admin, no para el técnico.

04 Integración OT ↔ Facturación

Conversión automática de items de OT a líneas de factura, con integración AFIP.

1

OT pasa a estado FACTURACIÓN

El admin genera la factura desde la OT. Se crea un registro Sale con estado BORRADOR y se vincula a work_orders.sale_id.

2

Conversión de items

Cada work_order_item se convierte en un SaleItem:

Tipo OT ItemLínea de Factura
REPUESTO[SKU] [Descripción] × [cant] = $[subtotal]
MANO_DE_OBRAMano de obra: [desc] [horas]h × $[tarifa/h] = $[subtotal]
SERVICIO_EXTERNOServicio externo: [desc] = $[precio]
3

Emisión del comprobante

Se cambia el estado de la Sale a EMITIDA. La factura va a la cola AFIP para obtener el CAE (Código de Autorización Electrónico).

4

Respuesta AFIP

Éxito: Se guarda CAE + fecha de vencimiento. Sale queda como EMITIDA con comprobante fiscal válido.

Fallo: Sale queda en cola de reintentos con backoff exponencial (1min, 5min, 15min, 1h). Alerta al admin.

5

Pago y cierre

Cuando el cliente paga, la Sale pasa a PAGADA y la OT a CERRADO.

🚫 Regla fiscal: Las facturas emitidas NUNCA se editan ni eliminan. Si hay error → se emite Nota de Crédito (NC) que anula parcial o totalmente.

05 Arquitectura de Sincronización Offline-First

Custom Event Sourcing + CRDT Híbrido. Diseñado para evolucionar la infraestructura existente (IndexedDB + fetch API) sin dependencias externas.

🚫 Alternativas descartadas

PouchDB

Requiere CouchDB como backend. No compatible con PostgreSQL existente. Migración de infraestructura completa. Descartado.

PowerSync

SaaS de pago con SDK propietario. Requiere SQLite nativo (WASM). No compatible con IndexedDB existente. Lock-in. Descartado.

ElectricSQL

Proyecto inmaduro (pre-1.0). Requiere wa-sqlite. API inestable. Riesgo alto para producción. Descartado.

WatermelonDB

Diseñado para React Native. Soporte web limitado y experimental. No es web-first. Descartado.

✅ Recomendación: Evolución de infraestructura existente

1. UUID v7 para IDs

Reemplazar auto-increment con UUID v7 en tablas transaccionales. Contiene timestamp → ordenable cronológicamente. Eliminan conflictos de ID entre dispositivos offline.

2. Event Log inmutable

Cada mutación genera un evento con: UUID v7, tipo, payload (before/after/delta), device_id, user_id, timestamp. Los eventos son la fuente de verdad para sync.

3. Estrategias por tipo de dato

No existe un "one size fits all". Cada tipo de dato usa la estrategia CRDT más apropiada según sus patrones de conflicto.

📊 Matriz de estrategias por tipo de dato

Tipo de Dato Estrategia Merge Conflicto? Resolución
Stock (counters) PN-Counter CRDT Σ incrementos por device − Σ decrementos por device Posible: stock negativo Alerta admin. Ventas no se anulan.
Ventas / Facturas Append-only Solo crear nuevos registros, nunca editar No (UUID v7 único por device) N/A — anular con Nota de Crédito
Órdenes de Trabajo LWW-Register per field Cada campo: gana timestamp más reciente Posible: 2 mecánicos editan misma OT Último en escribir gana. Historial conserva ambas versiones.
Config / Precios Server-wins Valor del servidor siempre prevalece No Server es source of truth
Clientes Field-level merge Merge campo a campo, flag conflictos reales Posible: datos divergentes Resolución manual por admin

06 Event Log Architecture

Tabla central de eventos inmutables que alimenta la sincronización, auditoría y replay.

📋 Tabla sync_events

CREATE TABLE sync_events (
  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  -- UUID v7 contiene timestamp → orden natural = cronológico
  event_type            VARCHAR(50) NOT NULL,
  -- STOCK_DECREMENT, SALE_CREATE, OT_UPDATE, OT_ITEM_ADD, etc.
  entity_type           VARCHAR(50) NOT NULL,
  entity_id             VARCHAR(50) NOT NULL,
  payload               JSONB NOT NULL,
  -- { before: {...}, after: {...}, delta: {...} }
  device_id             VARCHAR(50) NOT NULL,
  user_id               INT NOT NULL,
  company_id            INT NOT NULL,
  local_id              INT,
  created_at            TIMESTAMP NOT NULL,
  synced_at             TIMESTAMP, -- NULL = pendiente de sync
  server_processed_at   TIMESTAMP, -- NULL = no procesado aún
  conflict_resolved     BOOLEAN DEFAULT FALSE,
  conflict_resolution   TEXT
);

🔄 Flujo de sincronización

📱 DEVICE (offline)
Operación del usuario
↓
Evento creado con UUID v7
↓
Guardado en IndexedDB outbox
↓
Estado local actualizado (optimistic)
── WiFi ──→
Recibe ACK del servidor
↓
Marca evento como synced ✅
🖥️ SERVER
Recibe batch de eventos
↓
Valida integridad y permisos
↓
Detecta conflictos (LWW / CRDT)
↓
Aplica a PostgreSQL
↓
Guarda en sync_events
── Broadcast ──→
Envía ACK al device origen

↓

✅ Idempotencia: Cada evento tiene UUID único. Si un device re-envía un evento ya procesado, el servidor lo ignora (dedup por UUID).

07 PN-Counter CRDT para Stock

Positive-Negative Counter: cada dispositivo mantiene su propio contador de incrementos y decrementos. El servidor los combina sin conflicto.

📐 Ejemplo: Producto "Tornillo M8" (stock inicial: 20)

Device A — Local Centro

Incrementos (ingresos) 0
Decrementos (ventas/OT) 3
Stock local percibido 17

Device B — Local Norte

Incrementos (ingresos) 0
Decrementos (ventas/OT) 2
Stock local percibido 18

🔀 Server Merge

Total incrementos = max(A.inc, B.inc) = 0
(ingresos se procesan server-side, no per-device)
Total decrementos = A.dec + B.dec = 3 + 2 = 5
(decrementos son per-device, aditivos)
Stock real = stock_inicial + server_ingresos − Σ(device_decrementos)
Stock real = 20 + 0 − 5
= 15 ✅

⚠️ Escenario de conflicto: Stock negativo

Producto "Parche Gore-Tex" — Stock: 1

Device A (offline)

Vende 1 unidad −1
Stock local 0

Device B (offline)

Vende 1 unidad −1
Stock local 0

🔀 Server Merge — Conflicto

Stock = 1 − 1 − 1
= −1 ⚠️
→ ALERTA: Stock negativo para "Parche Gore-Tex"
→ Admin decide: reordenar al proveedor, transferir de otro local, o ajustar inventario
→ Las ventas NO se anulan — el cliente ya se llevó el producto

🏗️ Estructura IndexedDB (client-side)

// store: stock_counters
{
  "product_variant_id": "uuid-v7-xxx",
  "device_id": "device-abc",
  "increments": 0,        // solo ingresos locales
  "decrements": 3,        // ventas + OT items
  "last_server_stock": 20, // último valor conocido del server
  "last_sync": "2026-01-15T10:30:00Z"
}

// stock percibido = last_server_stock + increments - decrements
// 20 + 0 - 3 = 17
