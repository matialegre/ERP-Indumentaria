01 — Dimensionamiento

50 vs 200 vs 500 usuarios — ¿Qué cambia?

La tecnología base es la misma. Lo que cambia es el tamaño del servidor y componentes adicionales para manejar la concurrencia.

Componente 50 usuarios 200 usuarios 500 usuarios
Workers de API 4 8–12 16–24 (o 2 servidores)
RAM del servidor 16 GB 32–64 GB 128 GB (o distribuido)
CPU 8 cores 16–32 cores 32–48 cores
Conexiones DB Pool de 20 Pool de 80–100 Pool de 200 + réplica lectura
PgBouncer Opcional Obligatorio Obligatorio
Redis 1 instancia 1 instancia 1 instancia + Redis Cluster si es necesario
Arquitectura 1 servidor 1 servidor 2+ servidores (load balancer)
Base de datos 1 PostgreSQL 1 PostgreSQL 1 primario + 1 réplica de lectura
Internet 100 Mbps 500 Mbps 1 Gbps simétrico
Costo mensual (VPS) ~$65 USD ~$120–180 USD ~$350–500 USD
¿Qué es PgBouncer? — Cuando muchas personas hacen consultas a la base de datos al mismo tiempo, PostgreSQL abre una conexión por cada una, consumiendo memoria. PgBouncer se pone adelante y reutiliza un pool fijo de conexiones. A partir de ~100 usuarios es necesario.

Cómo se escala progresivamente

El sistema no necesita arrancar sobredimensionado. Se empieza con lo mínimo y se agregan recursos a medida que se necesitan. Cada paso se hace sin tirar lo anterior.

1
Arranque — 1 a 50 usuarios VPS de $30-65/mes

Un solo servidor VPS chico (8 cores, 16 GB). Todo corre junto: Nginx + FastAPI + PostgreSQL + Redis. Sin PgBouncer. Suficiente para los primeros meses de uso real.

No se tira nada al escalar: se migra la base de datos al servidor más grande.

2
Crecimiento — 50 a 200 usuarios VPS de $120-180/mes

Se cambia a un VPS más grande (16-32 cores, 32-64 GB). Se agrega PgBouncer. Se aumentan los workers de FastAPI a 8-12. La misma arquitectura, más recursos.

En Hetzner, cambiar de plan toma 5 minutos. No hay migración compleja.

3
Límite de 1 servidor — ~200-300 usuarios Techo del servidor único

Acá se llega al límite de lo que un solo servidor puede manejar bien. Las señales: tiempos de respuesta suben a >500ms en horarios pico, CPU constantemente >80%, queries lentas en la base de datos.

La solución no es poner un servidor más grande sino separar componentes.

4
Arquitectura distribuida — 300 a 500+ usuarios $350-500/mes (múltiples servidores)

Se separan los servicios en servidores distintos:

  • • Servidor 1: Nginx + Frontend (load balancer)
  • • Servidor 2: FastAPI workers (la API)
  • • Servidor 3: PostgreSQL primario (escrituras)
  • • Servidor 4: PostgreSQL réplica (lecturas — reportes, consultas de stock)
  • • Servidor 5 (opcional): Redis + PgBouncer + workers async

El código no cambia. Solo cambia dónde corre cada componente.

¿Cuándo se necesita escalar?

No se escala preventivamente, se escala cuando se detectan estas señales:

Señales de que falta capacidad

  • • Tiempos de respuesta > 500ms en horarios pico
  • • CPU del servidor constantemente > 80%
  • • RAM al 90%+ de uso
  • • Queries de base de datos tardan > 1 segundo
  • • Usuarios reportan "lentitud" al mediodía

Qué se hace en cada caso

  • • CPU alta → agregar más workers o más cores
  • • RAM alta → subir plan del VPS (minutos)
  • • Queries lentas → agregar réplica de lectura
  • • Todo lento → separar DB en su propio servidor
  • • 300+ usuarios → load balancer + 2 APIs
Dato clave: El código del sistema es el mismo en todos los pasos. Escalar no implica reescribir nada. Solo implica cambiar dónde y con cuántos recursos corre cada componente. Se puede arrancar con $65/mes y llegar a 500 usuarios con $500/mes sin tocar una línea de código.

Si el objetivo es llegar a 500 usuarios

El camino correcto es arrancar chico en infraestructura pero desde el día 1 escribir el código preparado para escalar. Esto no cuesta más plata ni más tiempo — es simplemente cómo se programa.

Lo que se hace desde el día 1 en el código (gratis)

  • ✓ Usar PostgreSQL desde el arranque (no SQLite)
  • ✓ API stateless: no guardar sesiones en memoria del servidor
  • ✓ Redis como caché desde el principio
  • ✓ Índices en la base de datos en todas las consultas frecuentes
  • ✓ Variables de entorno para todas las conexiones (DB, Redis, API URL)
  • ✓ Paginación en todas las tablas/listados (nunca cargar 10.000 registros)
  • ✓ Queries optimizadas: no SELECT *, no N+1, usar JOINs
  • ✓ Separación frontend/backend: React sirve archivos estáticos, API es independiente

Estas prácticas ya están contempladas en el stack propuesto (FastAPI + PostgreSQL + Redis + React). No agregan costo ni complejidad.

🌐

Nginx

El único proceso expuesto a internet. Maneja HTTPS (seguridad), comprime respuestas para ahorrar ancho de banda, distribuye carga entre workers y bloquea ataques básicos (DDoS, inyección).

⚛️

React + Vite + Tailwind

La interfaz que ven los usuarios. Se descarga UNA VEZ al navegador y después solo viajan datos (JSON). No recarga la página entera en cada clic — es como una app nativa.

🐍

FastAPI (Python)

El cerebro del sistema. Recibe pedidos, ejecuta lógica de negocio (stock, precios, facturación) y responde con datos. Async nativo = puede atender cientos de pedidos al mismo tiempo con pocos recursos.

🐘

PostgreSQL

La base de datos más robusta del mundo open source. Soporta escrituras simultáneas sin bloqueos (MVCC), índices para búsquedas de stock por talle/color, y particionamiento para millones de movimientos históricos.

⚡

Redis

Memoria ultrarrápida para datos que se consultan todo el tiempo: stock actual, listas de precios, sesiones de usuario. Evita golpear la base de datos por cosas que no cambian cada segundo.

🔀

PgBouncer

Se pone entre FastAPI y PostgreSQL. En vez de abrir 200 conexiones (una por usuario), reutiliza un pool de 80 conexiones. Esto reduce el uso de memoria de PostgreSQL de ~4GB a ~800MB.

04 — Desarrollo local

¿Puedo arrancar a desarrollar desde mi PC?

Tu PC actual de 16GB RAM: sí se puede para DESARROLLAR, pero con limitaciones. No es para producción.

Tu PC actual (16 GB RAM)

RAM disponible para desarrollo
~10-12 GB libres después de Windows + apps → suficiente para VS Code + Postgres + FastAPI + navegador
Capacidad para usuarios simultáneos
1-3 usuarios de prueba. NO producción.
Velocidad de compilación frontend
Aceptable con Vite (hot reload < 1 segundo)

Flujo recomendado

DESARROLLO Tu PC de 16GB

Escribís código, probás en local con datos de prueba. VS Code + Postgres local + navegador.

STAGING VPS chico (~$10/mes)

Servidor de pruebas. Los usuarios de confianza prueban antes de que sea oficial.

PRODUCCIÓN VPS grande (~$120/mes)

El sistema real. 200 usuarios. Solo se sube código probado.

El desarrollo se puede hacer desde una PC común. Para programar no se necesita un servidor potente — el servidor es necesario para cuando los 200 usuarios estén conectados. El código se escribe en local y se sube al servidor cuando está listo.
Limitación con 16 GB de RAM: Postgres local + VS Code + Chrome consumen ~6-8 GB. Windows ya usa 4-5 GB. Quedan ~3 GB de margen. Funciona para desarrollo, pero justo. Con 32 GB sería más cómodo.

¿Qué es Figma?

Es la herramienta estándar de la industria para diseñar interfaces. Es gratuita para equipos de hasta 3 personas. Las diseñadoras crean ahí el "plano" visual completo del sistema, y el programador lo construye siguiendo esos planos exactos.

Las diseñadoras NO necesitan aprender a programar. Solo necesitan Figma.

Sistema de diseño — lo que se define antes de programar

🎨 Paleta de colores

Colores primarios de la marca, secundarios, neutros (grises), y semánticos (verde = OK, rojo = error, amarillo = advertencia). Se define una vez y se usa en TODO el sistema.

🔤 Tipografía

1 o 2 fuentes máximo. Inter (la que usa esta página) es el estándar actual para apps. Se definen tamaños para títulos, subtítulos, cuerpo y datos de tabla.
